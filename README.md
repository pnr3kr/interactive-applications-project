## Design Justification
Our sketches evolved out of our milestone through the necessity of introducing complexity in the design while maintaining our core design principles that ensures our app functions appropriately. Our guiding principles in terms of the persona and design principles remained the same. We have in some cases needed to make a tradeoff in reducing clarity in order to convey information appropriately, such as with the visualization tab. We opted to create a widget for the main display on the home page so that information could easily be conveyed on startup. For the app page we created a widget that let the user select the day so that the user could have the flexibility to select which forecast they wanted to see with minimal clutter. The adding an event widget is so that users can quickly and comprehensively schedule a day where they may need a unique outfit. This layout supports the primary task of being able to enter record special events to include when planning outfits beforehand, while also being able to edit events and delete them if needed. Finally the visualization widgets were placed next to each other to minimize scrolling. Our primary task for the user is to be able to immediately see the outfit planned for them on that day, while also giving the option for a week's forecast. We believe layout supports this because the daily outfit is the first thing viewed on the app, and the forecast is a click away onto a separate tab. Our chart types were selected because we believe that a line chart (for temperature) and a pie chart (for outfit category) are best at communicating this type of information. We believe giving the user a view of average temperature allows for the user to understand what kind of clothes they may need to have ready (maybe they need to clean clothes/buy new clothes). Furthermore, the pie chart allows the user to identify holes in their wardrobe. If they need dress clothes three days a week and only have a few dress shirts, that's an indicator for the user that they may need a few more dress shirts.Our initial sketches were a little oversimplified compared to the final version. 

We underestimated how difficult some of the implementation would be, and as such needed to add a little more visual clutter to fully convey the meaning of the app. We primarily had to trade aesthetics for the functionality of the app. While a large calendar may not be visually appealing, we felt it necessary so that the user could have a comprehensive view of their schedule across the entire month, and also allow them to have full freedom in scheduling their events. Another tradeoff was the ability to import a calendar, such as Google Calendar or a phone app calendar like apple. Similarly, the implementation for this would be very complex, needing to allow Google Calendar API to connect, which wasn't very beginner-user friendly due to its complex process.

## Visual Design Review
### Detail view: 
One thing that worked well is consistent typography. The font sizes clearly separate the daily outfit (largest), section headers (medium), and supporting details like weather (smaller). This supports readability and reinforces hierarchy. However, we noticed an issue with spacing consistency. Some calendar cells had tighter padding than others, making the layout feel uneven. We fixed this by standardizing margins and padding across all elements so each section feels balanced. Another improvement involved alignment. A few icons and text labels were slightly misaligned within cards. We adjusted them to align with a consistent grid, which made the interface look cleaner and more professional.
### Page-level view:
The app does a good job with visual hierarchy. The daily outfit is the first thing users see, which matches our goal of helping users quickly decide what to wear. The weekly calendar is secondary, which keeps the focus clear. One issue we found was clutter, especially on pages with the full calendar and visualizations. There was not enough whitespace, making it harder to scan information. We improved this by adding more spacing between sections and slightly reducing the size of less important elements. We also confirmed that the content fits properly. There were minor cases where text in calendar cells felt cramped, so we adjusted sizing to prevent overflow and improve readability.
### Function-level view:
The app successfully handles success states. For example, when a user adds an event, the outfit updates immediately, giving clear feedback that the action worked. However, we identified missing loading and error states. For example, when weather data updates, there was no clear loading indicator. We noted this as an area for improvement and would add a loading icon or message. We also saw that user feedback could be stronger. Some interactions (like selecting a day) did not clearly show that the selection was active. We improved this by adding highlights or visual changes when a user clicks something.
### App-level view:
Across the app, design patterns are mostly consistent. Buttons, cards, and layout structures follow the same style, which helps users learn the interface quickly. This matches our goal of consistency and ease of use. One issue we found was inconsistency between pages, especially in how information is grouped. Some pages felt more crowded than others. We worked to standardize layouts so all pages follow a similar structure. Navigation is generally clear and logical, with the home screen focused on today’s outfit and additional information available through other sections. However, we noted that some features (like visualizations) feel slightly disconnected. This could be improved by better integrating them into the main flow of the app.
